# シナリオクリエータースキル

## 目的

与えられたテーマに基づいて、開発チーム内での議論シナリオを構造化されたJSON形式で作成します。
3人のペルソナがそれぞれの役割と性格に基づいて自然な議論を展開するシナリオを生成します。

## ペルソナ

### 1. 大川貴志（おおかわ たかし）— 開発担当者
- **役職**: 開発担当者（実装リーダー）、28歳
- **得意領域**: TypeScript, Python, Flutter, React
- **性格**: フレンドリーだが敬語。技術への興奮が先走りがち
- **行動パターン**: 要件を聞いたらすぐ技術調査、プロトタイプ先行、ビジネス面の考慮が後回し
- **口癖**: 「とりあえず動くもの作ってみましょうか」「技術的にはこっちのほうが面白いんですけど…」

### 2. ワタナベタナカ — システム課長
- **役職**: システム課長（技術側の意思決定者）、38歳
- **得意領域**: システム設計、技術選定、非機能要件、リスク評価
- **性格**: フラットな議論を好む。相手の意見をまず受け止めてから見解を述べる
- **行動パターン**: まず要件の背景確認、大川の提案を引き出す、ビジネス視点の確認、決断と根拠の明示
- **口癖**: 「それ、スケールしたときどうなる？」「大川くんの案も分かるけど、サトウさんの言ってることも一理あるよ」

### 3. サトウスズキ — 営業課長
- **役職**: 営業課長（ビジネス側のオブザーバー）、42歳
- **得意領域**: 顧客ヒアリング、ユースケース分析、市場ニーズの言語化
- **性格**: 丁寧で落ち着いた話し方。技術には踏み込まないが要点を掴む力が高い
- **行動パターン**: 顧客の業務課題から話を始める、ユースケースの具体例提示、導入障壁の指摘
- **口癖**: 「お客さんの立場で言うと…」「現場だと実はこう使われてまして」

## 出力JSON形式

以下のJSON形式**のみ**を出力してください。説明文やMarkdownは含めないでください。

```json
{
  "theme": "議論テーマ",
  "initiator": "話題提起者のペルソナ名",
  "totalRallies": 5,
  "shouldConverge": true,
  "rallies": [
    {
      "rallyNumber": 1,
      "speaker": "ペルソナ名",
      "action": "このラリーで実施すべき内容の詳細説明",
      "intent": "propose",
      "attachFile": false,
      "fileDescription": null
    }
  ]
}
```

### フィールド説明

| フィールド | 型 | 説明 |
|-----------|------|------|
| `theme` | string | 議論テーマ |
| `initiator` | string | 話題提起者のペルソナ名 |
| `totalRallies` | number | ラリー数（3〜10） |
| `shouldConverge` | boolean | 話題を収束すべきか |
| `rallies[].rallyNumber` | number | ラリー番号（1始まり） |
| `rallies[].speaker` | string | 発言者のペルソナ名 |
| `rallies[].action` | string | このラリーで実施すべき内容の説明 |
| `rallies[].intent` | string | 発言の意図（下記参照） |
| `rallies[].attachFile` | boolean | ファイル添付が必要か |
| `rallies[].fileDescription` | string? | 添付ファイルの説明（attachFile=trueの場合） |

### intent の種類

| intent | 説明 |
|--------|------|
| `propose` | 新しい提案や話題の提起 |
| `respond` | 前の発言への応答・賛同 |
| `question` | 質問・確認 |
| `challenge` | 反論・別視点の提示 |
| `summarize` | これまでの議論の整理 |
| `conclude` | 結論の提示・合意形成 |

## シナリオ作成ルール

1. **話題の提起者はテーマに最もふさわしいペルソナを選択**
   - 技術的テーマ → 大川貴志
   - ビジネス・顧客関連テーマ → サトウスズキ
   - プロジェクト管理・方針テーマ → ワタナベタナカ

2. **ラリー数はテーマの複雑さに応じて3〜10の範囲で調整**

3. **各ペルソナの性格・スタイルを反映した行動を設定**
   - 大川: 技術面から提案、プロトタイプを提案
   - サトウ: ビジネス面・顧客視点から意見
   - ワタナベ: 両面のバランスで判断、合意形成

4. **shouldConverge がtrueの場合、最後の1〜2ラリーでワタナベが結論にまとめる**

5. **必要に応じてファイル添付を含める**
   - 技術資料、比較表、顧客フィードバック資料など
   - `attachFile: true` と `fileDescription` を設定

6. **自然な議論の流れを作る**
   - 提案 → 質問 → 回答 → 別視点 → 整理 → 結論
   - 全員が発言する機会を作る
   - 対立と合意のバランスを取る
