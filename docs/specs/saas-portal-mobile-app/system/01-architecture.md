# システム構成・技術スタック

## システム構成図

```
エンドユーザー (Android / iOS)
    ↓
Kotlin + Jetpack Compose (Android)
Swift + SwiftUI (iOS) [Phase 1]
    ↓  Bearer Token (JWT)
FastAPI (認証 / API / データ取得)
    ↓
MongoDB 7
```

PoC段階ではAndroid版を先行開発し、バックエンドは既存のFastAPI + MongoDB構成を活用する。社内管理向けアプリ（saas-management-app）とはユーザーDBを分離し、ポータル専用の認証基盤を構築する。

## レイヤーごとの責務

| レイヤー | 主な責務 |
|---|---|
| モバイルアプリ | ログイン/サインアップ、利用状況ダッシュボード表示、チャート描画、アプリ起動 |
| バックエンド | JWT認証、テナント別認可、利用状況データ取得API、アプリ起動エンドポイント |
| データベース | テナント・ユーザー・利用実績データの保持 |

## 技術スタック

### Mobile（Android）— PoC

| 項目 | 技術 |
|---|---|
| 言語 | **Kotlin** |
| UIフレームワーク | **Jetpack Compose** |
| チャートライブラリ | Rechartsライクなネイティブ向けチャートライブラリ（選定予定） |
| HTTP通信 | Retrofit / Ktor Client |
| 認証 | JWT トークン管理 |

**選定理由（大川氏）：** 「Androidが Kotlin+Jetpack Compose、iOSはSwift+SwiftUIで組むのが今だと王道」

### Mobile（iOS）— Phase 1

| 項目 | 技術 |
|---|---|
| 言語 | **Swift** |
| UIフレームワーク | **SwiftUI** / **UIKit**（必要に応じて） |
| チャートライブラリ | Swift Charts / 同等ライブラリ |

### Backend

| 項目 | 技術 |
|---|---|
| フレームワーク | **FastAPI** |
| 言語 | **Python 3.12** |
| スキーマ / バリデーション | **Pydantic v2** |
| MongoDBドライバ | **Motor** |
| パッケージ管理 | **uv** |

**選定理由（大川氏）：** 「バックエンドは今のFastAPI+MongoDBの構成をそのまま活かせるので、API層はかなりスムーズにいける」

### Database

| 項目 | 技術 |
|---|---|
| データベース | **MongoDB 7** |
| データ特性 | テナント・ユーザー・利用実績を格納するドキュメント指向DB |

### 認証

| 項目 | 技術 |
|---|---|
| 認証方式 | **JWT認証**（自前実装） |
| パスワードハッシュ | **passlib / bcrypt** |
| JWT実装 | **python-jose** |
| トークン管理 | アクセストークン + リフレッシュトークン |

## アーキテクチャ上の前提

### 認証・認可

- 社内管理向けアプリとは**完全にユーザーDBを分離**し、ポータル専用のサインアップ・ログインフローを用意する
- テナントの下にユーザーが紐づく構造（テナント → ユーザー）
- JWTベースの認証で、既存のsecurityモジュールをベースに拡張する
- トークンリフレッシュの仕組みも含めて実装する

### API設計方針

- 既存のFastAPI構成をベースに、ポータル向けAPIを追加
- テナントスコープでのデータアクセス制御（ユーザーは自テナントのデータのみ参照可能）
- RESTful API設計に準拠

### 開発アプローチ

- **PoC:** Android版先行、3機能（認証・ダッシュボード・Mock起動）に絞る
- **段階的展開:** Android版で検証後、iOS版に展開
- **Mock実装:** PoC段階ではアプリ起動はMock。ディープリンク連携はPhase 1

## 技術選定の経緯（Teams議論より）

- **大川氏:** モバイル側はAndroidがKotlin+Jetpack Compose、iOSはSwift+SwiftUIで組むのが今だと王道。バックエンドは今のFastAPI+MongoDBの構成をそのまま活かせる
- **大川氏:** テナントの下にユーザーが紐づく構造で、JWTベースの認証を考えている。社内管理向けとは完全にユーザーDBを分離
- **大川氏:** データ可視化はRechartsライクなネイティブ向けチャートライブラリを選定する予定
- **ワタナベPM:** いきなり両OS同時は重いので、まずAndroid版で検証してからiOSに展開する段階的アプローチ
