# 認証・権限設計・運用フロー

## 認証フロー

![認証・権限設計](../images/image6.png)

```
ログイン画面
  → FastAPI 認証エンドポイント
  → JWTアクセストークン発行
  → 保護API呼び出し
  → 各APIで JWT と users.is_active を検証
```

### 認証方式

- FastAPI側で **自前のJWT認証基盤** を構築する
- 当初は Entra ID SSO 連携も検討されたが、MVPでは立ち上がり速度を優先して JWT 自前実装を採用する
- Phase 1 のアクセストークンは **60分有効** とし、ブラウザの **sessionStorage** に保持する
- 永続的なログイン維持は行わず、トークン期限切れ時は再ログインを求める
- `localStorage` には保存しない

### アカウントポリシー

- パスワードは `passlib / bcrypt` でハッシュ化して保存する
- 管理者のみユーザーを作成 / 無効化できる
- 無効化されたユーザーは次回APIアクセス時点で利用不可とする
- 連続ログイン失敗時は管理者確認を要する運用とし、監査ログに記録する

## ロール設計

2つのロールで権限を分離する。

| 機能 | sales | admin |
|---|---|---|
| ダッシュボード閲覧 | 担当顧客のみ | 全件 |
| 顧客 / 契約編集 | 担当顧客のみ | 全件 |
| マスタ管理 | 閲覧不可 | CRUD |
| 従量データ取込 | 実行可（担当領域） | 実行可（全件） |
| 取込の再開 / 締め日超過対応 | 不可 | 可 |
| ユーザー管理 | 不可 | 可 |
| 監査ログ閲覧 | 不可 | 可 |

### 認可の考え方

- 営業担当者のアクセス範囲は `customers.assigned_sales_user_id` で判定する
- 契約、月次実績、ダッシュボードは顧客に紐付けて権限制御する
- 直接URLアクセスやAPI直叩きでも、サーバ側で必ず認可チェックを行う

## 従量課金データ更新フロー

![従量課金データ更新フロー](../images/sekkei_image4.png)

### 月次運用フロー（毎月1-5営業日）

```
SaaS製品管理画面
  → CSVダウンロード（営業担当者が手動実行）
  → システムにCSVアップロード
  → バリデーション（対象月 / マスタ / 重複 / 数値範囲）
  → プレビュー確認
  → 保存・確定
  → ダッシュボード反映
```

### 運用ルール

| 項目 | 内容 |
|---|---|
| 更新頻度 | 月次（毎月1〜5営業日目、JST） |
| 更新担当 | 営業担当者（各SaaS製品からCSVダウンロード） |
| 締め後の再取込 | 管理者のみ許可 |
| Phase 1方式 | CSV手動取込 |
| Phase 2方式 | SaaS製品APIからの自動データ取得 |
| 同一月の再取込 | 置換として扱い、`usage_imports` に履歴を残す |
| ダッシュボード更新タイミング | 確定後に反映 |

### エラー時の扱い

- バリデーションエラー時は `monthly_usage` を更新しない
- 担当営業がCSVを修正して再アップロードする
- 失敗履歴は `usage_imports.status = failed` として残す
- 管理者は取込履歴と監査ログから再実行可否を判断する

### 月中の単価変更対応

サトウ氏の指摘：「現場だと月中に単価変更が入ることもある」

Phase 1 では以下の方針とする。

- 単価変更は `contract_plan_history` に必ず記録する
- 自動按分は行わず、月次確定前に管理者が最終請求値を確認する
- 自動按分ロジックは Phase 2 の拡張要件として扱う

## 管理者運用

- ユーザーの追加 / 無効化 / 権限変更は管理者のみ実施
- 主要な CRUD、取込確定、再取込、ユーザー無効化は `audit_logs` に記録する
- マスタ削除は物理削除ではなく `is_active = false` を基本とする

## テスト / リリース運用

- 主要な業務フロー（ログイン、契約更新、CSV取込、ダッシュボード確認）は Playwright でE2Eテストを用意する
- 本番リリース前にステージング環境で移行リハーサルを実施する
- 初回リリース時は Excel 元データとの件数 / 金額突合レポートを作成する

## Phase 2: API自動連携

将来的にCSV手動取込を廃止し、SaaS製品APIから直接データを取得する。

- 月次バッチまたはWebhook連携でCSV手動取込を廃止
- 契約変更履歴をもとに月中単価変更の自動按分を実装
- SSO連携や運用自動化は Phase 2 以降の検討対象とする
