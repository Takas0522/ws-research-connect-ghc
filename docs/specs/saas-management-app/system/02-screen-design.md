# 画面構成・機能要件

## 画面構成案

![画面構成案](../images/sekkei_image2.png)

大川氏が提案した4画面構成。ワタナベPMの「とりあえず①と④あたりからプロトを作る」方針に基づき、ダッシュボードと従量データ取込を優先開発する。

## 1. ダッシュボード画面

### 概要

営業が日常的に確認するメイン画面。顧客ごとのAPI利用率を一覧表示し、超過リスクをアラートで通知する。

### 画面要素

| 要素 | 内容 |
|---|---|
| 利用状況サマリ | 顧客別API利用率の一覧表示（カード形式） |
| 超過アラート一覧 | 使用率閾値（例: 90%）超過時の警告バッジ |
| 月次トレンドグラフ | Rechartsによる利用推移のライングラフ |
| 利用率チャート | 全顧客横断の利用率チャート |

### ユースケース

1. 営業担当者がログイン後、ダッシュボードで担当顧客の利用状況を確認
2. 超過アラートが出ている顧客を確認し、プラン見直し提案を検討
3. 利用推移グラフ（サトウ氏要望：利用者数の推移・利用目的の傾向）でデータドリブンな提案材料を取得

## 2. マスタ管理画面

### 概要

管理者が製品・プラン・メトリクスの基本情報を登録・管理する画面。一覧 → 詳細 → 編集のフロー。

### マスタテーブル

大川氏が提案した4つのマスタ：

| マスタ | 内容 | 主な項目 |
|---|---|---|
| 製品マスタ | SaaS製品の基本情報 | 製品名、カテゴリ、ベンダー |
| 課金プランマスタ | プラン別の料金・上限値 | プラン名、月額基本料、API上限、ストレージ上限 |
| 顧客マスタ | 契約主体の情報 | 顧客コード、顧客名、担当者 |
| メトリクス定義マスタ | 従量項目定義 | API呼出数、ストレージ等の従量指標 |

> メトリクス定義マスタがあると、将来的に新しい課金指標が増えても柔軟に対応できる（大川氏）

## 3. 契約管理画面

### 概要

顧客ごとの契約内容・ステータスを管理する画面。

### 機能

| 機能 | 内容 |
|---|---|
| 契約一覧 | 全顧客の契約状況を一覧表示（フィルタ・検索可能） |
| 契約詳細 | 契約開始日・更新日・月額基本料・ライセンス数の表示・編集 |
| 更新時期アラート | 契約更新日が近い顧客を通知 |
| プラン変更履歴 | ライセンス数・プランの変更履歴を時系列で表示 |
| 契約ステータス管理 | 契約中・更新手続中・解約等のステータス遷移 |

## 4. 従量データ取込画面

### 概要

月次の従量課金データをCSVで取り込む画面。Phase 1ではCSV手動アップロード、Phase 2でAPI自動連携を想定。

### 機能

| 機能 | 内容 |
|---|---|
| CSVアップロード | ドラッグ&ドロップまたはファイル選択でCSVをアップロード |
| 取込結果プレビュー | アップロードしたデータの事前確認表示 |
| バリデーション | 整合性チェック（マスタとの突合・重複チェック・数値範囲チェック） |
| エラーチェック・差分確認 | エラー内容の表示、前月との差分ハイライト |

### 運用フロー（ワタナベPM確認事項）

「④の従量データ取込、プロトに入る前に運用面を詰めておきたい。誰がいつ更新するのか、月次バッチなのか手動アップロードなのか、そこが曖昧だと作り直しになる。」（ワタナベPM）

→ サトウ氏の回答：「現場だと各SaaS製品の管理画面から月初にCSVをダウンロードして、担当者が手動でExcelに転記。更新は月次で、毎月5営業日目くらいまでに前月分をとりまとめる。ここが属人化していて引き継ぎが難しい。」
