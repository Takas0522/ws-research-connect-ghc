# 非機能要件

## 性能要件

### レスポンスタイム

| 操作 | 目標 |
|------|------|
| 一覧取得（100件以下） | 200ms 以内 |
| 詳細取得 | 300ms 以内 |
| 登録・更新 | 500ms 以内 |
| ダッシュボード集計 | 1,000ms 以内 |
| 請求額計算（単件） | 100ms 以内 |
| 一括利用実績登録（50件） | 2,000ms 以内 |

### データ量の想定

| 項目 | 初期 | 1年後想定 |
|------|------|---------|
| 製品数 | 7 | 15 |
| プラン数 | 14 | 30 |
| 顧客数 | 7 | 50 |
| 契約数 | 15 | 150 |
| 月次利用実績（累計） | 45 | 1,800 |
| トライアル（累計） | 0 | 100 |

> 現時点での規模は小さいため、パフォーマンス要件は緩やかに設定する。
> インデックス設計で基本的なクエリ性能を確保する。

## セキュリティ要件

### v1 スコープ

- **認証・認可**: v1 では実装しない（社内ツールとしてローカル/VPN環境で利用前提）
- **入力バリデーション**: サーバーサイドで全入力をバリデーション
- **SQLインジェクション対策**: EF Core のパラメータバインディングで対策済み
- **XSS対策**: React の自動エスケープに依存
- **CORS**: フロントエンド開発サーバー（localhost:5173）のみ許可

### 将来対応（v2 以降）

- Azure AD / Microsoft Entra ID 連携
- ロールベースアクセス制御（RBAC）
- HTTPS 強制
- アクセスログ記録

## データ整合性

### トランザクション管理

以下の操作はトランザクション内で実行する。

| 操作 | トランザクション範囲 |
|------|-------------------|
| プラン変更 | 契約更新 + 変更履歴作成 |
| 契約解約 | 契約更新 + 変更履歴作成 |
| トライアル転換 | トライアル更新 + 契約作成 |
| 利用実績登録 | UPSERT + 請求額計算 |
| 一括利用実績登録 | 全件を1トランザクションで処理 |

### データ保全

- 物理削除は行わない（全てステータスによる論理管理）
- `updated_at` はトリガーで自動更新
- 契約変更は必ず ContractHistory に記録

## 可用性

- v1 はローカル開発環境での利用が主
- Docker Compose で PostgreSQL を起動する構成
- 本番環境へのデプロイは v2 以降で検討

## テスト要件

### ユニットテスト

| 対象 | フレームワーク | カバレッジ目標 |
|------|-------------|-------------|
| バックエンド（課金計算ロジック等） | xUnit | ビジネスロジックは80%以上 |
| フロントエンド（コンポーネント） | Vitest + Testing Library | 主要コンポーネント |

### 統合テスト

| 対象 | フレームワーク |
|------|-------------|
| API エンドポイント | xUnit + WebApplicationFactory |

### E2E テスト

| 対象 | フレームワーク |
|------|-------------|
| 主要ユーザーフロー | Playwright + Testcontainers |

### テスト対象の優先度

1. **必須**: 請求額計算ロジック（ユニットテスト）
2. **必須**: 契約 CRUD + プラン変更 + 解約（統合テスト）
3. **推奨**: ダッシュボード集計 API（統合テスト）
4. **推奨**: トライアルのライフサイクル（統合テスト）
5. **推奨**: 主要画面フローの E2E テスト

## ログ・監視

### v1 スコープ

- ASP.NET Core の標準ロギング（`ILogger`）
- 開発時は `Development` レベルで SQL クエリも出力
- ヘルスチェック: `/healthz`（既存）

## 国際化・ローカライゼーション

- v1 は日本語のみ
- 通貨は日本円（¥）固定
- 日付は `YYYY-MM-DD` 形式（ISO 8601）
- タイムゾーンは JST（Asia/Tokyo）を基準とするが、DB 格納は UTC
