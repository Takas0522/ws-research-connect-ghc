# システム構成・技術スタック

## システム構成図

![システム構成概要](../images/image3.png)

```
営業担当者 / 管理者 (ブラウザ)
    ↓
React + TypeScript SPA
    ↓  Bearer Token
FastAPI (認証 / API / 取込処理)
    ↓
MongoDB 7
```

Phase 1では社内利用を前提としたシンプルな3層構成とし、月次CSV取込で更新されたデータをダッシュボードに反映する。リアルタイム同期ではなく、**月次バッチ更新を前提にした運用システム**として設計する。

## レイヤーごとの責務

| レイヤー | 主な責務 |
|---|---|
| フロントエンド | ログイン、画面遷移、入力バリデーション、チャート表示、CSVアップロード |
| バックエンド | JWT認証、ロール別認可、業務バリデーション、MongoDB CRUD、CSV取込処理 |
| データベース | マスタ、契約、月次実績、取込監査、監査ログの保持 |

## 技術スタック

### Frontend

| 項目 | 技術 |
|---|---|
| フレームワーク | **React 19** |
| 言語 | **TypeScript 5.7** |
| ビルド | **Vite 6** |
| ルーティング | **React Router v7** |
| UI | **Tailwind CSS** |
| グラフライブラリ | **Recharts** |

**選定理由:** ダッシュボード中心の画面構成とフォーム入力が多く、SPAでの高速な画面遷移とコンポーネント再利用がしやすい。

### Backend

| 項目 | 技術 |
|---|---|
| フレームワーク | **FastAPI** |
| 言語 | **Python 3.12** |
| スキーマ / バリデーション | **Pydantic v2** |
| MongoDBドライバ | **Motor** |
| パッケージ管理 | **uv** |

**選定理由:** 非同期I/Oと型付きスキーマで、CSV取込やマスタ更新APIを比較的少ないコード量で安全に実装できる。

### Database

| 項目 | 技術 |
|---|---|
| データベース | **MongoDB 7** |
| データ特性 | 契約・従量実績・監査ログの混在を扱うドキュメント指向DB |

**選定理由:** マスタ / 契約 / 月次実績 / 監査ログなど粒度の異なるデータを1つのアプリケーション内で管理しやすい。

### 認証 / 品質

| 項目 | 技術 |
|---|---|
| 認証方式 | **JWT認証** |
| パスワードハッシュ | **passlib / bcrypt** |
| JWT実装 | **python-jose** |
| E2Eテスト | **Playwright** |

## アーキテクチャ上の前提

### 認証・認可

- Phase 1は FastAPI 側で JWT 認証を実装する
- ロールは `sales` と `admin` の2種類とする
- 営業担当者の参照範囲は `customers.assigned_sales_user_id` を基点に API 側で絞り込む
- Entra ID SSO 連携は将来検討事項とし、Phase 1の必須要件には含めない

### 運用・更新方式

- 従量データは毎月1〜5営業日（JST）に CSV で取り込む
- 同一月の再取込は置換として扱い、`usage_imports` に履歴を残す
- ダッシュボードには最終更新日時を表示し、データが月次更新であることを明示する

### 非機能要件

- 日時は **UTC保存 / JST表示** を基本とする
- ダッシュボード初期表示は **P95 2秒以内** を目標とする
- CSVアップロード上限は **10MB** を目安とする
- 主要APIの応答時間は **P95 500ms以内** を目標とする
- ステージング環境での確認と移行リハーサルをリリース前提条件に含める

## 技術選定の経緯

Teams議論での検討ポイント：

- **ワタナベPM:** 社内用アプリなので、当初はEntra ID SSO連携も検討したい
- **大川氏:** FastAPI側でまずJWT認証基盤を用意した方が、プロトタイプとMVPの立ち上がりが早い
- **結論:** MVPでは FastAPI + JWT認証を採用し、ユーザー管理・権限制御を先に成立させる

フロント React + バックエンド FastAPI + MongoDB の構成は、大川氏が「ダッシュボードと取込フローを同時に作りやすい」と評価している。
