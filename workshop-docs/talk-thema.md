## 2. 会話のテーマ

### 2-1. 最近"効いた"AI活用／失敗したAI活用

- CLIやM365 CopilotのResearchモードの活用
  - 参加者①④の期待に直結：MS365内ドキュメントをResearchモードで横断検索し、開発コンテキストに取り込む実例
  - 事前議論の論点：セマンティックサーチ vs エージェンティックサーチ — 単なるキーワード検索ではなく、**目的・ゴールに応じて複数ソースを横断する検索**をどう実現するか
- 名づけのお話
  - 変数名・関数名の命名をCopilotに委ねるケース、うまくいく場合と意図がズレる場合
- WorkIQでどこまでできるか
  - 開発者の生産性可視化と、AI活用効果の測定
  - 事前議論で畠山がWorkIQ活用中と共有済み — 具体的な知見を掘り下げる
- **設計書からのコード生成**（Workshop本編の核）
  - 実際の設計書持ち込みによるコード生成試行が予定されている
  - 参加者の実プロジェクト（Flutter/SwiftUI/Vue.js/React等）でどこまで精度が出るか
  - 事前議論の方針：既存コードの移行より**ドキュメントベースで新規開発するアプローチ**を推奨（畠山）
  - ポイントカードのサンプルドキュメントを用意予定
- **既存コードベースの理解支援**
  - 参加者②の課題：チーム開発で既存コードを素早く理解し、機能追加につなげるフロー
- **GitHub Copilotのコードレビュー活用**
  - 事前議論で畠山がレビュー機能の有効性を共有：セキュリティチェック・コード修正提案の自動化
  - PR作成→Copilotレビュー→修正の実運用フロー

### 2-2. ボトルネックになっているもの

- レビューなんかをどのような観点で行うか
  - AI生成コードのレビュー基準：人が書いたコードと同じ基準でよいのか
  - セキュリティ・品質担保の観点
- Localで実行するもの、Cloudで実行するもの
  - GitHub Copilot Agent のローカル実行 vs クラウド実行の使い分け
  - モバイル開発（Flutter/Swift/Kotlin）特有のビルド環境制約
- **タスク分割の粒度**
  - 参加者②の課題：Copilotに渡すタスクの適切な粒度とは
  - 大きすぎると精度が落ち、小さすぎるとオーバーヘッドが増える
  - 粒度分割も含めAIの仕事にしても良い
- **マルチ言語・マルチFW環境での運用**
  - 参加者②③④はプラットフォーム横断（Web + Mobile + Backend）
  - 言語・FWごとにCopilotの効き方が異なる問題
- **CLI/SDKによるフロー制御**
  - 事前議論で共有：エンタープライズ向けアプリでは自動化フローの各フェーズを**決定論的に区切る設計**が必須
- **プロンプト設計力のギャップ**
  - 事前議論で組織的課題として認識済み
  - カスタムエージェント開発時のシステムプロンプト設計の難しさ

### 2-3. データとAI

- データの収集と管理
- **社内ナレッジとCopilotの接続**
  - MS365（SharePoint, Teams, OneNote等）に蓄積されたドキュメントをCopilotのコンテキストとして活用
  - 参加者③④の期待：社内ナレッジをコーディングに反映させるワークフロー
  - 事前議論の課題：情報が総務・Slack・メール・Teams等に分散しており、**チャットベースの情報は名前付けが不十分で発見性が低い**
- **設計書のデータとしての価値**

  - 設計書の構造化度合いがコード生成精度にどう影響するか
  - 事前議論の方針：コード移行ではなく**ドキュメントベースの新規開発**を推奨
